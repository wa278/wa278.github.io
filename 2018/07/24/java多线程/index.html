java多线程

发表于 2018-07-24

线程安全性

核心在于管理共享的，可变的状态
当多个线程访问一个可变的变量时，如果没有合适的措施，就会产生错误，解决方案有两个：

  • 修改状态为不可变对象
  • 在访问变量时使用同步机制

原子性操作

不正确的进行原子性操作，容易发生竞态条件（Race Condition）,常见的竞态条件有如下两种：

  • a++操作 实际上包含了三步：读，加，写。
  • 先检查后执行，比如延迟初始化，if(a == null)
    要确保单个原子操作中更新所有有关的状态变量

用锁来保护对象

  • 同一个对象的读取都要同一把锁保护。
  • 同一个不变性条件涉及的对象，都要用同一把锁。

可见性

很多种原因使得一个线程看不到其他线程修改变量的结果(重排序等)。JMM(java内存模型规定了JVM必须遵守的一组最小保证(happens-before)，规定了写入操作在何时对其他线程可见)。
happens-before:

  1. 程序顺序规则：同一个线程A操作在B操作之前，A对B是可见的。这是指当B有用到A的时候，不会发生重排序，没用到的时候，也可能发生重排序。只确保了逻辑上是对的，实际怎么做，是否重排序不一定。和单线程程序一样。
  2. 监视器锁规则：加锁操作能看到之前解锁操作的全部内容，是可见的。解锁操作在同一个锁加锁之前执行。
  3. volatile规则：volatile读写能看到之前的内容，写操作在读操作之前执行。
  4. 线程启动规则：Thread.start 必须在该线程任何操作之前执行
  5. 线程结束规则：任何操作都在线程结束之前运行。
  6. 中断规则：调用Interrupt 必须在检测到中断之前执行
  7. 终结器规则： 构造函数必须在启动终结器之前完成
  8. 传递性 A hb B , B hb c , 则 A hb C.

解决方式：

  1. 上锁
  2. volatile变量

不可变对象

不可变对象的赋值操作是原子的，所以不可变对象是线程安全的。
不可变对象需要满足以下条件：

  • 对象创建后器其状态就不能修改（不提供修改的方法）
  • 对象的所有域都是final类型
  • 对象都是正确创建的，this指针不能溢出。

可以使用volatile 来发布不可变对象，这样就保证了可见性。

事实不可变对象

如果对象从技术上来看是可变的，但其状态在发布后不会变化（不提供修改的方法），称为事实不可变对象，程序只需将它们视为不可变对象即可。 不可变对象需要放在final修饰词下发布。

线程封闭

引起多线程问题的根源就是 共享和可变，不可变对象解决了可变，线程封闭解决了共享。Swing和JDBC都使用了线程封闭技术。

  1. Ad-hoc 封闭 不推荐使用
  2. 栈封闭 只通过局部变量才能访问对象，局部变量只有此线程此函数才能访问，所以是安全的。其他线程无法访问。
  3. ThreadLocal类 提供了get,set 方法，为每个线程都保留一个相同名称但是独立的变量。Threadlocal 本质是一个Map<Thread,T>。

发布与溢出

在构造函数中发布对象时，可能导致 this引用隐式溢出，因为其他的对象的方法可能会调用this对象的方法，而this还没有构造完成。可以使用工厂方法先创建对象，再发布对象，而不是在构造函数中发布。

安全发布

安全发布就是既不让引用溢出，又让所有线程都要看到这个对象（此对象的可见性）。安全发布的常用模式：

  1. 使用静态初始化static,stack在类初次加载时创建，只存在主内存中，所有线程都可以看到。
  2. 用volatile 和 原子变量 atomic
  3. 保存在某个正确构造的final域类型中，即事实不可变对象。
  4. 保存在有所的容器里例如 hashtable, synchronizedMap, ConcurrentMap,CopyOnWriteArrayList, CopyOnWriteArraySet, SynchronizedList,BlockingQueue,ConcurrentLinkedQueue
